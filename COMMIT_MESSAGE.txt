Fix catastrophic recruiter migration failure (99.6% data loss)

CRITICAL FIX: Recruiter migration system was failing silently, with only
24 out of 6,137 recruiters (0.4%) being migrated successfully.

Root Cause:
-----------
The FK mapping file (ProjectId.json) was never created automatically during
project migration. When recruiter migration tried to map ProductStock.ProductId
to project.Id, it found no mappings and silently skipped 6,113 rows (99.6%).

This happened because:
1. Project migration captured idMappings in memory but never wrote them to file
2. No validation before recruiter migration to check FK mappings exist
3. Poor reporting - "24/6137 rows" didn't distinguish skipped vs errors

Impact:
-------
- 6,113 recruiters LOST (99.6%)
- Donation attribution completely broken
- Recruiter tracking system non-functional
- Historical recruiter data essentially lost

Changes Made:
-------------

1. AUTO-CREATE ProjectId.json (src/server.js:2697-2727)
   - During project migration, immediately save idMappings to ProjectId.json
   - Happens BEFORE connections close and memory is cleared
   - Logs sample mappings for verification
   - Result: Complete FK mapping file auto-generated every time

2. ADD VALIDATION before recruiter migration (src/server.js:751-828)
   - STEP 2.5: New validation step before STEP 3
   - Checks ProjectId.json exists (fails fast if missing)
   - Validates ALL ProductIds from ProductStock have mappings
   - Warns about missing mappings and calculates impact
   - Shows which ProductIds are missing (first 10)
   - Result: Early warning system prevents silent failures

3. IMPROVE REPORTING (src/server.js:591-674, 860-869)
   - Track 3 metrics separately: Inserted / Skipped / Errors
   - Log WHY rows were skipped (e.g., "Missing FK mapping: ProjectId=1957")
   - Group skip reasons by type and show counts
   - Clear breakdown at end of each migration step
   - Result: Full visibility into what happened

Lessons Learned:
----------------
1. Never trust silent success - always validate results
2. Test FK mapping files BEFORE dependent migrations
3. Distinguish skipped vs errors in reporting
4. Never assume IDs are preserved (AUTO_INCREMENT!)

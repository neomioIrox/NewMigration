# מדריך הרצה — "הרצה מלאה" (Pipeline)

מדריך תפעול ידני לפיצ'ר ההרצה המלאה. נבנה 2026-07-15; ספק ותוכנית ב-`docs/superpowers/`.

## 1. הכנות

1. **שרת:** `cd server && npm start` (פורט 3001). בעלייה השרת יוצר אוטומטית את טבלאות המעקב ומסמן כ-failed ריצת pipeline שנקטעה בקריסה קודמת.
2. **קליינט:** `cd client && npm run dev` (פורט 5173).
3. לגלוש אל: **http://localhost:5173/pipeline** (קישור "הרצה מלאה" בתפריט הצד).

> חלופה בלי UI — דרך ה-API:
> ```
> POST http://localhost:3001/api/pipeline/start   גוף: {"mode":"continue"} או {"mode":"fresh"}
> POST http://localhost:3001/api/pipeline/stop
> GET  http://localhost:3001/api/pipeline/current
> GET  http://localhost:3001/api/pipeline/runs
> ```

## 2. מה יש בדף

- **checkbox "התחל מאפס"** —
  - **לא מסומן (ברירת מחדל = המשך):** אם הריצה האחרונה נכשלה/נעצרה — ממשיכים אותה בדיוק מהשלב שנעצר (הכפתור יציג "המשך מהנקודה שנעצרה"). אם אין ריצה פתוחה — נוצרת ריצה חדשה.
  - **מסומן (מאפס):** ריצה חדשה של כל 20 השלבים מהשלב הראשון. **שימו לב: "מאפס" לא מוחק שום דבר מהיעד** — ניקוי היעד הוא משימה נפרדת שטרם הוגדרה. מיועד לטבלאות יעד ריקות.
- **הכפתור הראשי** — מזניק את הריצה. מושבת בזמן ריצה. לחיצה כפולה/דפדפן שני יקבלו שגיאת "כבר רץ" (409).
- **עצור** — עוצר בעדינות: השלב הנוכחי מקבל pause מהמנוע, הריצה מסומנת "נעצר". ממשיכים אחר-כך עם אותו כפתור (בלי לסמן מאפס).
- **רשימת 20 השלבים** — סטטוס לכל שלב (✓ הושלם / ⟳ רץ / ○ ממתין / ✗ נכשל), מונים (עובדו/הוכנסו/דולגו/שגיאות), פס התקדמות חי לשלב הרץ, והודעת שגיאה מלאה בשלב שנכשל.
- רענון דפדפן באמצע ריצה בטוח — הדף נטען מהמצב השמור ב-DB.

## 3. סדר השלבים

Affiliate → Source → CustomerUser → LutFundCategory → 5 מיפויי פרויקטים (Funds, Collections, Type2, Type3 אבות→בנים) → Prayer → FundCategory → ProjectItemLocalization → RecruitersGroup → Recruiter → גלריות (תמונות → מדיה → וידאו) → **Donation** → PrayName → AsakimDonation.

הסדר מוגדר ב-`server/config/pipeline.json` (כולל תלויות ו-batchSize פר שלב) ומאומת בטעינה — שגיאת קונפיגורציה תיחסם עוד לפני שנוצרת ריצה.

## 4. סמנטיקת "המשך" — חשוב להבין

בלחיצה על המשך, לכל שלב שלא הושלם המערכת בודקת את ריצת המנוע הקודמת שלו:

| מצב הריצה הקודמת של השלב | מה קורה |
|---|---|
| הסתיימה (completed) | השלב מסומן כהושלם ומדולג |
| הושהתה יפה (paused — עצרתם עם "עצור") | **המשך אמיתי מאותה שורה** דרך מנגנון ה-resume הקיים — בלי כפילויות |
| נקטעה לא-נקי (קריסה/שגיאה) עם שורות שכבר נכתבו | **הריצה נעצרת עם הודעה** — המשך עיוור היה יוצר כפילויות ביעד. נדרש טיפול ידני בשלב הזה (ניקוי/השלמה) לפני המשך |
| לא התחילה / לא כתבה כלום | מתחיל רגיל |

> ההגבלה על שלב שנקטע לא-נקי היא זמנית — פיצ'ר ה-Checkpoint (ספק `2026-07-15-migration-checkpoint-design.md`) ייתן המשכיות פר-mapping גם במקרים אלה.

## 5. כשל באמצע

שלב שנכשל ⇒ כל הריצה נעצרת מיד, השלב אדום עם הודעת השגיאה, שאר השלבים נשארים "ממתין". לאחר תיקון הבעיה — לוחצים "המשך מהנקודה שנעצרה". קריסת שרת באמצע ⇒ בעלייה הבאה הריצה מסומנת failed וממשיכים באותו אופן.

## 6. אזהרות

- **לא להשתמש במסך `/migrate` הישן בזמן שה-pipeline רץ** (ולהפך) — אין נעילה הדדית בין שני המנגנונים, והרצה כפולה של אותו שלב משני המקומות תכתוב פעמיים לאותה טבלה.
- בפרט: אם ה-pipeline עצר שלב, הריצה המושהית שלו תופיע גם ברשימת "מיגרציות מושהות" ב-`/migrate` — **לא** להמשיך אותה משם; להמשיך רק מכפתור ה-pipeline.
- מומלץ בהרצה הראשונה: לעצור יזום באמצע שלב גנרי קטן (למשל Affiliate/Source) וללחוץ המשך — לוודא שה-resume עובד מול מנוע אמיתי לפני שמגיעים לשלבים הכבדים (Donation ~שעות).

## 7. איפה המצב נשמר

`migration_tracker` (MySQL מקומי): `pipeline_runs` (ריצה), `pipeline_run_steps` (שלב-בריצה, מקושר ל-`migration_runs` של המנוע). היסטוריית ריצות: `GET /api/pipeline/runs`.
